# 팀원용 피쳐 설명서 — 또래 강사 마켓플레이스를 위한 합성 콘텐츠 파이프라인

> 이 문서는 **이 피쳐를 처음 보는 팀원**이 30분 안에 "우리가 뭘 만들고 있고, 왜 이렇게 만드는지" 를 이해할 수 있게 쓴 설명서입니다.
> 동시에 **졸업 논문 / 학회 논문의 메인 기술**로 저술할 때 필요한 배경·용어·선행 연구도 함께 담았습니다.
> 전제: 읽는 사람이 머신러닝·분산 시스템 배경 지식이 없다고 가정합니다.

---

## 0. 한 줄 요약

> **"실제 사용자가 0명인 신규 플랫폼을 위해, 사람처럼 행동하는 가상 유저 50명을 시뮬레이션으로 돌려 '자연스러운 행동 로그'를 만들고, 그 로그를 바탕으로 LLM이 피드·후기·메시지를 쓰게 한 뒤, 다층 검증을 통과한 것만 서비스 DB에 publish 하는 시스템."**

한 마디로 "가짜인데 진짜 같은" 초기 콘텐츠를 **재현 가능한 파이프라인으로** 뽑아내는 장치입니다.

---

## 1. 무슨 문제를 푸는가 (Why)

### 1-1. 우리 제품의 정체성

우리가 만드는 서비스는 **또래 강사 마켓플레이스**입니다.

- 호스트(supporter)는 프로 강사가 아니라 "나 기타 칠 줄 아는데" 수준의 또래입니다.
- 목적: 용돈 벌이 + 취미 공유 + 친구 만들기.
- 파트너(participant)는 "비싼 수업 말고 편하게 배우고 싶다"는 또래입니다.
- 톤: "알려드릴게요"(X) / "같이 해볼래요"(O)

예: _"수원 화서동 기타 같이 쳐볼 분, 1:1 5천원, 저희 집 거실에서 해요"_

### 1-2. 신규 플랫폼의 근본 고민 — Cold Start

신규 서비스는 출시 시점에 **사용자도 0명, 후기도 0개, 피드도 비어 있음** 상태입니다. 그런데 유저가 첫 방문 시 피드가 비어 있으면 바로 이탈합니다. 추천 시스템 학계에서는 이를 **Cold-Start Problem**이라고 부르는 아주 전통적인 문제입니다.

보통 해결책은 세 가지인데 전부 한계가 있습니다.

| 기존 해결책                                  | 한계                                                              |
| -------------------------------------------- | ----------------------------------------------------------------- |
| 실제 유저가 쓸 때까지 기다린다               | 유저가 안 오니 아무도 안 씀 (닭-달걀)                             |
| 운영팀이 수기로 가짜 피드를 만든다           | 편향·비용·확장성 문제. 10개는 쓰지만 1,000개는 못 씀              |
| LLM에게 "피드 1,000개 써줘" 라고 직접 시킨다 | 내용이 전부 비슷해지고, 지역·가격·관계 맥락이 없어 "가짜 티"가 남 |

### 1-3. 우리의 가설

> **"사람을 직접 만들지 말고, 사람의 *행동*을 먼저 만든 뒤, 그 행동을 LLM이 글로 옮겨 쓰게 하자."**

이게 이 피쳐의 핵심 아이디어이자 논문의 메인 주장입니다.

- 행동은 **에이전트 기반 시뮬레이션(Agent-Based Simulation)** 으로 만든다.
- 글쓰기는 **LLM(거대 언어 모델)** 에게 맡긴다.
- 단, LLM이 거짓말을 지어내지 못하게 **여러 층의 검증 필터**를 통과시킨다.

### Stage 1. 실제 지역 데이터 수집·정규화 — `local-context-builder`

#### 어떤 기술인가

**지역의 "분위기 벡터"를 숫자로 만들어두는 작업**입니다.

카카오 Local API에서 수원시 특정 동네(예: 화서동)의 카페·식당·공원 같은 실제 장소(POI, Point of Interest)를 대량으로 긁어옵니다. 이걸 그대로 쓰면 노이즈가 많으니, "이 동네는 카페 밀도 60%, 밤 시간 활동 지수 0.7" 같은 **요약 숫자(feature vector)** 로 정규화합니다.

비유: 동네마다 **MBTI 같은 프로파일**을 붙여두는 작업. 시뮬레이터가 "이 에이전트는 화서동에 산다"고 하면, 그 동네가 어떤 동네인지 일관되게 알 수 있습니다.

#### 왜 이렇게 판단했는가

- **완전 가상의 지역을 쓰면 현실감이 0**. "서울 강남역" 같은 실제 이름이 있어야 독자가 몰입합니다.
- 실제 POI 데이터 전체를 끌고 다니면 너무 무겁습니다. 정규화 층을 통해 "시뮬에 필요한 최소 피처"만 추출합니다.
- 이 단계는 **한 번만 돌리면 됨** (지역 데이터는 하루 단위로 거의 안 바뀜). 그래서 batch job으로 분리했습니다.

#### 선행 연구 / 참고 기술

- **Urban computing / POI embedding** — Zheng et al., _"Urban Computing: Concepts, Methodologies, and Applications"_ (ACM TIST, 2014). 지역 피처를 숫자로 요약하는 기초 프레임워크.
- **Location-based social network research** — Foursquare·Yelp 계열 POI 데이터 활용 추천 연구 다수.
- **Point-of-Interest recommendation surveys** — Liu et al., _"A Survey of POI Recommendation"_ (2017).

논문에서 이 단계는 "**데이터 수집 및 전처리(Data Acquisition & Feature Engineering)**" 섹션으로 묶으면 됩니다.

### Stage 2. 또래 강사 에이전트 시뮬레이션 — `spot-simulator`

#### 어떤 기술인가

**가상의 유저 50명을 컴퓨터 안에서 일주일 동안 살게 하는 장치**입니다.

각 가상 유저(agent)는

- **페르소나**: 성향 (예: `night_social`, `homebody`, `weekend_explorer` 등 5종)
- **스킬**: 기타 Lv.3, 홈베이킹 Lv.4 같은 "가르칠 수 있는 것 / 배우고 싶은 것"
- **자산**: 지갑 잔고, 주중/주말 여유 시간, 장비(기타·요가매트 등), 집 공간 유무, 평판
- **관계**: 지금까지 함께 수업한 다른 유저들과의 친밀도

를 가지고 있습니다. 시뮬레이터는 시간을 **tick** 단위(1 tick ≈ 반나절)로 흘려보내면서, 매 tick마다 각 에이전트에게 "너 지금 뭐 할래?"를 확률 공식으로 물어봅니다.

예:

- _"너(spontaneous 페르소나)는 기타 Lv.3, 용돈 동기 0.85, 지금 토요일 오후, 피로 30%. → **집에서 기타 1:1 수업을 5천원에 열 확률 0.42** → 주사위 굴려서 ✅ → `CREATE_TEACH_SPOT` 이벤트 기록"_

이런 식으로 48 tick × 여러 날을 돌리면 `event_log.jsonl` 파일에 **수만 건의 이벤트가 시간 순서대로 쌓입니다**. 이게 Stage 3의 입력입니다.

#### 핵심 공식 몇 개 (논문 수식 블록에 그대로 쓸 수 있는 것)

**호스트가 수업을 열 확률**

```
p_teach(s, t) =
    teach_appetite(s)            # 이 스킬 가르치고 싶은 동기
  × (skill_level(s) / 5)         # 실력
  × pocket_money_motivation       # 용돈 벌이 동기
  × (1 - fatigue / max_fatigue)   # 피로도
  × time_availability(t)          # 주중/주말 여유
  × space_capability(s)           # 공간 가능 여부
  × region_create_affinity(r, s)  # 지역별 스킬 수요
```

**파트너가 참여할 확률**

```
p_learn(s, t) =
    learn_appetite(s)
  × (1 - skill_level(s) / 5)      # 이미 잘하면 배울 동기 낮음
  × budget_capability(fee)        # 지갑 대비 수업료
  × (1 - fatigue / max_fatigue)
  × time_availability(t)
```

**단골 보정**

```
p_join_bonded = 2.0     if relationship ∈ {regular, mentor_bond, friend}
              = 1.0     otherwise
```

이 공식들은 "사람이 행동할 때 고려할 법한 요인"을 곱셈 형태로 조합한 것으로, 추후 **각 가중치를 고정하고 재현 가능한 시드로 돌리면 같은 결과**가 나옵니다. 논문 재현성 확보에 중요합니다.

#### 왜 이렇게 판단했는가

**대안 A: 실제 유저 행동 로그를 그대로 쓴다**
→ 우리는 실제 유저가 없음. 탈락.

**대안 B: LLM에게 직접 "수업 모집글 1,000개 써줘"**
→ LLM이 쓰는 글은 평균값에 수렴해서 다양성이 망함. 지역·시간·인물 간 **일관성**이 없음. 예를 들어 "김수지가 어제 기타 수업을 열었다"는 문장과 "김수지가 오늘 요가 수업을 열었다"는 문장이 서로 모순될 수 있음.

**대안 C (우리 선택): 시뮬레이션으로 먼저 "사실"을 만들고, LLM은 그 사실을 문장으로 옮기기만 한다**
→ LLM은 **글쓰기**(문체·감정·톤)만 담당. **사실**(누가, 언제, 어디서, 얼마에)은 시뮬이 결정.
→ 덕분에 LLM의 hallucination이 "사실 수준"에서 차단됨.
→ 논문 기여점 #1: **"LLM을 문서 생성기로 격하시키고, 사실 생산은 결정론적 시뮬에 맡긴다"** — 이 분리가 이 연구의 핵심 주장입니다.

#### 선행 연구 / 참고 기술

- **Agent-Based Modeling (ABM)** — Epstein & Axtell, _"Growing Artificial Societies"_ (1996). Schelling의 분리 모델(1971)까지 거슬러 올라가는, 사회과학 시뮬레이션의 기초.
- **Generative Agents: Interactive Simulacra of Human Behavior** — Park et al. (Stanford, 2023). LLM으로 25명의 가상 인물을 작은 마을에 살게 한 연구. 우리 접근과 직접 비교 가능. **차이점**: 그들은 LLM을 행동 결정에 썼고(느리고 비쌈), 우리는 행동 결정은 확률 공식으로 하고 LLM은 글쓰기에만 씀(빠르고 재현 가능).
- **PersonaChat / Persona-based dialogue** — Zhang et al., _"Personalizing Dialogue Agents"_ (ACL, 2018). 페르소나 개념을 대화에 주입한 계보.
- **Multi-agent simulation for market design** — Tesauro & Das, _"High-Performance Bidding Agents"_ 등 ABM 기반 시장 시뮬 연구 계보.
- **Cold-start recommendation** — Schein et al., _"Methods and Metrics for Cold-Start Recommendations"_ (SIGIR, 2002). 우리가 해결하려는 원 문제.

### Stage 3. 매력 큐레이션 (Multi-Run Attractiveness Curation)

#### 어떤 기술인가

**시뮬레이션을 여러 번 돌려서 대량의 로그를 만든 뒤, "혹할 만한" 스팟만 골라내는 필터**입니다.

시뮬은 공짜이기 때문에(LLM 호출 없음, 그냥 Python), 조건을 바꿔가며 5~10번씩 돌립니다:

- Run 1: 평균 분포 (baseline)
- Run 2: 호스트 비율을 높인 버전
- Run 3: 주말 활동 집중 버전
- Run 4: 특정 지역 몰린 버전
- ...

그러면 약 1,500개의 "가상 수업(spot)" 기록이 쌓이는데, 여기서 **"유저가 진짜 보면 클릭할 것 같은"** 스팟만 골라냅니다. 기준(매력 신호 8종) 예:

- 단골 관계로 전환된 케이스
- 첫 호스팅인데 대성공
- 용돈 벌이 사례
- 관계 전환 이벤트 (mentor_bond → friend)
- 협상(counter-offer)을 거친 케이스 등

#### 왜 이렇게 판단했는가

- **"현실적 분포 ≠ 매력적 분포"** — 시뮬이 현실적으로 돌아가면 노쇼 10%, 매칭 실패 40% 같은 분포가 나옵니다. 그대로 피드에 올리면 신규 유저가 이탈합니다. 초기 피드는 **큐레이션된 매력**이 필요합니다.
- **"선별이지 조작이 아니다"** — 로그 자체를 수정하지 않습니다. 시뮬이 만든 사실(fact)은 그대로 두고, **어떤 사실을 보여줄지만** 고릅니다. 이 원칙은 논문에서 데이터 조작 윤리 의심을 차단하는 데 중요합니다.
- **다양성 보장** — 상위 N개만 뽑으면 전부 비슷해지므로, diversity-aware selection(코사인 유사도 기반 MMR과 유사)으로 스팟 간 중복을 피합니다.

#### 선행 연구 / 참고 기술

- **MMR (Maximal Marginal Relevance)** — Carbonell & Goldstein, _"The Use of MMR, Diversity-Based Reranking for Reordering Documents"_ (SIGIR, 1998). 다양성을 고려한 선별의 고전.
- **Curriculum / selective sampling** in ML — 좋은 샘플을 먼저 보여주는 전략.
- **Offline evaluation with synthetic logs** — Belletti et al. (Google), Yao et al. 등. 시뮬로 만든 로그로 추천 시스템을 사전 평가하는 연구 계열.

### Stage 4. LLM 콘텐츠 생성 + 6-Layer 검증 — `synthetic-content-pipeline`

#### 어떤 기술인가

**선별된 스팟 로그를 입력으로 받아, LLM에게 5종 콘텐츠를 쓰게 하고, 6단계 필터로 통과시키는 장치**입니다.

5종 콘텐츠:

1. **Feed preview** — 메인 피드 카드 한 줄
2. **Spot detail** — 상세 페이지 본문
3. **Plan** — 수업 진행 계획
4. **Messages** — 호스트↔파트너 간 대화 메시지 4종
5. **Review** — 수업 후기 별점 + 본문

각 콘텐츠는 **후보 2개(candidate × 2)** 씩 생성합니다. 하나가 검증에 떨어지면 다른 하나를 쓸 수 있도록.

#### 6-Layer 검증 파이프라인

```
Layer 1. 스키마 검증      JSON 구조가 맞는가? 필드가 누락됐나?
Layer 2. 룰 검증          금기어 포함? 수업료 상한 초과? 페르소나 톤 이탈?
Layer 3. 교차 참조        스팟 내 여러 콘텐츠가 서로 모순되지 않는가?
                            (feed에 "5천원"인데 detail에 "1만원"이면 reject)
Layer 4. LLM Critic       LLM이 다른 LLM의 결과를 평가 (sampling)
                            자연스러움·페르소나 일치·톤 점수화
Layer 5. 다양성 검증      n-gram·TF-IDF로 "비슷한 표현 반복" 감지
Layer 6. 최종 스코어링    위 다섯 점수를 가중 합산
                            통과한 것만 DB publish
```

**실패하면 어떻게 되는가?**

- Critic이 "이 문장은 프로 강사 톤입니다" 같은 **feedback**을 주면, 그 feedback을 프롬프트에 붙여서 **재생성**합니다. 이걸 통과할 때까지 최대 N번 반복 (rejection feedback loop).

#### 왜 이렇게 판단했는가

**왜 "6 Layer"인가?**

- LLM 결과를 하나의 판정기로만 걸러내면 놓치는 게 너무 많습니다. 층을 겹쳐야 각 층이 잡는 오류 종류가 달라집니다 (**defense in depth**).
- Layer 1·2는 **결정론적·저렴함** → 먼저 돌림.
- Layer 4 Critic만 LLM을 또 호출하므로 비쌈 → **샘플링만** (전체의 20%).
- Layer 5는 **다양성**을 보장해서 "같은 문장이 반복되는 현상"을 사전 차단.

**왜 LLM을 직접 쓰지 않고 `codex exec` CLI를 subprocess로 부르는가?**

- 이 프로젝트는 연구 단계라 OpenAI/Anthropic API 비용을 쓰지 않습니다.
- 대신 사용자의 **Codex 구독(CLI)** 을 subprocess로 호출해 동일한 모델을 씁니다.
- 장점: 비용 0, 재현 가능 (동일 prompt → 동일 응답 캐시 활용).
- 단점: 응답 지연 존재. 그래서 Critic은 sampling.

**왜 후보 2개인가?**

- LLM은 확률적이라 한 번에 좋은 결과가 나오지 않을 수 있습니다.
- 2개 생성해두면 "하나가 탈락해도 다른 걸 쓴다"는 안전장치가 됩니다.
- 이건 **self-consistency**(Wang et al. 2023)의 아주 단순화된 버전입니다.

#### 선행 연구 / 참고 기술

- **LLM-as-a-Judge** — Zheng et al., _"Judging LLM-as-a-Judge with MT-Bench and Chatbot Arena"_ (NeurIPS, 2023). LLM이 다른 LLM 결과를 평가하는 패러다임의 원조. 우리 Layer 4 Critic이 이 계보.
- **Self-Refine** — Madaan et al., _"Self-Refine: Iterative Refinement with Self-Feedback"_ (NeurIPS, 2023). 생성 → 자기 피드백 → 재생성 루프. 우리 "rejection feedback loop"와 거의 동일.
- **Reflexion** — Shinn et al., _"Reflexion: Language Agents with Verbal Reinforcement Learning"_ (NeurIPS, 2023). 실패를 자연어 피드백으로 학습하는 계보.
- **Constitutional AI** — Bai et al. (Anthropic, 2022). 모델이 규칙을 참조해 자신의 출력을 비판하고 수정하는 방법론. 우리 룰 검증 + Critic 구조의 사상적 근거.
- **Chain-of-Verification (CoVe)** — Dhuliawala et al. (2023). LLM이 자기 출력의 사실성을 검증하는 기법.
- **Self-Consistency** — Wang et al., _"Self-Consistency Improves Chain of Thought Reasoning"_ (ICLR, 2023). 후보 2개를 뽑아 비교하는 아이디어의 근원.
- **Constrained text generation** — 제약 기반 텍스트 생성 연구 (금기어·스키마 강제) 계열.

## 4. 이 피쳐를 어디에 활용할 수 있는가 (제품 시나리오)

지금까지 §1~§3이 "무엇을 만들었는가"였다면, 이 섹션은 **"만들어둔 자산(시뮬 로그 + 합성 콘텐츠 + 매력 점수)을 실제 제품에서 어떻게 꺼내 쓸 것인가"** 입니다. 네 가지 시나리오 각각에 대해 **엔드포인트·요청/응답 DTO·프론트 사용 예시**를 붙였습니다. (DTO는 Pydantic / TypeScript 혼용 표기로, 필드명은 그대로 구현에 쓸 수 있게 작성했습니다.)

### 용어 정의: 가상 / 현실 / 혼합

| 모드        | 정의                                                                   | 데이터 소스                                              |
| ----------- | ---------------------------------------------------------------------- | -------------------------------------------------------- |
| **virtual** | 시뮬 에이전트가 만든 행동 로그를 LLM이 문서로 옮겨 쓴 합성 피드만 표시 | `synthetic_spot`, `synthetic_content` (Stage 2~4 산출물) |
| **real**    | 실제 사람이 만든 진짜 유저 피드만 표시                                 | `spot`, `content` (프로덕션 DB. 초기엔 비어 있음)        |
| **mixed**   | 사용자의 페르소나·취향에 맞춰 virtual + real 피드를 weighted blending  | 둘 다 + `persona_region_affinity`                        |

모든 엔드포인트 응답의 SpotCard에는 `provenance` 필드가 박힙니다. 프론트는 이 값으로 "AI 생성" 배지·CTA를 분기합니다.

```python
class Provenance(StrEnum):
    VIRTUAL = "virtual"          # 합성 100%
    REAL = "real"                # 실제 유저 100%
    MIXED_ORIGIN = "mixed_origin" # 실 스팟인데 설명 일부가 합성
```

### 시나리오 1 — 가상/현실/혼합 토글 맵 (취미 탐색 UX)

**사용자 행동**: 지도 화면에서 토글 버튼으로 `virtual | real | mixed` 전환. 내 페르소나에 맞는 스팟이 지도 위 핀으로 뜨고, 카드 리스트가 하단에 슬라이드로 올라옴.

#### API

```http
GET /api/v1/map/spots
  ?mode=virtual|real|mixed
  &bbox=37.2701,127.0065,37.3015,127.0452   # sw_lat,sw_lng,ne_lat,ne_lng
  &persona_id=spontaneous                    # 선택: 개인화
  &skill_filter=기타,드로잉                    # 선택: 스킬 필터
  &time_slot=weekend_afternoon                # 선택
  &limit=30
Authorization: Bearer <jwt>
```

#### Response DTO

```python
class MapSpotsResponse(BaseModel):
    mode: Literal["virtual", "real", "mixed"]
    total: int
    spots: list[SpotCard]
    blend_ratio: BlendRatio | None   # mode=mixed 일 때만

class SpotCard(BaseModel):
    spot_id: str
    provenance: Provenance
    title: str                        # "수원 화서동 기타 같이 쳐볼 분"
    skill_topic: str                  # "기타"
    teach_mode: Literal["1:1", "small_group", "workshop"]
    venue_type: Literal["cafe", "home", "studio", "park", "gym"]
    fee_per_partner: int              # 원화
    location: GeoPoint
    host_preview: HostPreview
    persona_fit_score: float          # 0~1, 내 페르소나 대비 적합도
    attractiveness_score: float       # 0~1, Stage 3 매력 점수
    feed_preview: str                 # 한 줄 피드 카피 (LLM 생성 or 실제)
    is_synthetic_cta_enabled: bool    # virtual 이면 True → §시나리오 2 연결

class BlendRatio(BaseModel):
    virtual_count: int
    real_count: int
    mixed_count: int
    rationale: str  # "실제 스팟이 3개뿐이라 합성 27개로 보강했어요"

class HostPreview(BaseModel):
    display_name: str
    persona_type: str                 # virtual 이면 "spontaneous" 같은 페르소나 키
    skill_level: int                  # 1~5
    is_virtual_persona: bool
```

#### 프론트 사용 예 (의사코드)

```ts
const { spots, mode, blend_ratio } = await api.get('/v1/map/spots', {
    mode: toggle, // 'virtual' | 'real' | 'mixed'
    bbox: mapBounds,
    persona_id: user.persona,
});

spots.forEach((spot) => {
    renderPin(spot);
    if (spot.provenance === 'virtual') {
        renderBadge(spot, '지금은 AI가 만든 예시 스팟');
        renderCTA(spot, '이 스팟을 직접 요청해보세요 →'); // 시나리오 2 진입
    }
});

if (mode === 'mixed' && blend_ratio) {
    renderToast(blend_ratio.rationale);
}
```

#### 구현 메모

- `mode=virtual`은 `synthetic_content` 테이블에서 현재 활성 `dataset_version`으로 스캔, 매력 점수 내림차순 + 지역·페르소나 필터.
- `mode=real`은 프로덕션 `spot` 테이블. cold-start 구간엔 결과 0건이 정상.
- `mode=mixed`는 `real_count` 먼저 채우고, `limit - real_count` 만큼 virtual로 보강. `persona_region_affinity` 가중치로 `virtual` 점수를 개인화.

### 시나리오 2 — "이 스팟을 직접 요청해보세요" (AI 피드 → 실 스팟 전환)

**사용자 행동**: 시나리오 1에서 본 AI 생성 스팟 카드의 CTA를 탭 → 스팟 생성 폼으로 이동. 폼에는 **플레이스홀더(제목·소개·플랜·가격)** 가 미리 채워져 있고, "시뮬 미리보기" 버튼으로 **내가 이 스팟을 열면 에이전트 팀이 어떻게 반응할지**를 확인할 수 있음.

이 시나리오는 두 개의 하위 API로 나눕니다.

- 결과의 `narrative`만 LLM으로 한 번 호출(요약). Codex 브리지 캐시 적용.

### 시나리오 3 — 내 스팟 매력도 리포트 (supporter용 대시보드)

**사용자 행동**: 호스트가 자기 스팟 상세 화면에서 "매력도 분석" 탭 → 점수·백분위·좋아할 페르소나 그래프를 확인.

#### API

```http
GET /api/v1/spots/{spot_id}/attractiveness-report
Authorization: Bearer <jwt>   # 호스트 본인만 허용
```

```python
class AttractivenessReport(BaseModel):
    spot_id: str
    composite_score: float            # 0~1
    percentile_in_region: float       # 0~100
    percentile_in_skill: float

    signals: AttractivenessSignals    # Stage 3 8가지 신호

    audience_fit: AudienceFit
    improvement_hints: list[Hint]

class AttractivenessSignals(BaseModel):
    title_hookiness: float            # 제목이 클릭을 유발하는 정도
    price_reasonableness: float       # 가격이 페르소나 지갑 대비 합리적인 정도
    venue_accessibility: float
    host_reputation_fit: float
    time_slot_demand: float
    skill_rarity_bonus: float         # 희귀 스킬이면 가산
    narrative_authenticity: float     # 또래 톤 유지 정도
    bonded_repeat_potential: float    # 단골 유입 잠재력

class PriceBenchmark(BaseModel):
    your_fee: int
    p25: int
    p50: int
    p75: int
    p90: int
    verdict: Literal["too_cheap", "competitive", "slightly_high", "too_high"]
    similar_spots: list[SpotCard]

class AudienceFit(BaseModel):
    top_personas: list[PersonaMatch]    # 이 스팟을 좋아할 페르소나 순위
    demographic_hint: str               # "주말 활동형 20대 여성 비중 높음"
    expected_likers_per_week: int

class PersonaMatch(BaseModel):
    persona_type: str
    match_score: float
    estimated_join_probability: float

class Hint(BaseModel):
    category: Literal["title", "price", "plan", "venue", "tone"]
    message: str                       # "제목에 지역명을 넣으면 +8% 기대"
    estimated_lift: float              # 예상 점수 상승
```

#### 프론트 사용 예

```ts
const report = await api.get(`/v1/spots/${spotId}/attractiveness-report`);

radarChart.render(report.signals); // 8축 레이더
priceBar.render(report.price_benchmark); // p25/50/75/90 + 내 위치
personaBars.render(report.audience_fit.top_personas); // 페르소나별 점수

    hintList.add(`${hint.message} (+${hint.estimated_lift * 100}%)`);
});
```

#### 구현 메모

- `signals`는 시나리오 1과 동일한 Stage 3 큐레이터 로직을 **단일 스팟에 대해 실행**한 결과.
- `price_benchmark`는 동일 `skill_topic` + 동일 `region_cluster` 내 `synthetic_spot` + `real_spot`의 fee 분포에서 산출. 샘플 수가 적으면 `skill_topic` 전국 분포로 fallback.
- `audience_fit`은 `persona_region_affinity` × 스팟 벡터 코사인 유사도.

### 시나리오 4 — 관전 모드 "페르소나 라이브" (시뮬 바이럴 콘텐츠)

**사용자 행동**: 로그인 안 한 유저도 볼 수 있는 공개 페이지. "지금 이 순간 우리 가상 동네에서는?" 같은 네러티브로, 50명 페르소나가 tick 단위로 움직이는 걸 실시간 재생·하이라이트 클립 공유. 틱톡/숏폼으로 뽑아 바이럴 소재로 활용.

#### API

**4-1. 실행된 시뮬 런 목록**

```http
GET /api/v1/simulation/runs?variant=baseline|high_engagement|weekend_peak&limit=10
```

```python
class SimulationRunSummary(BaseModel):
    run_id: str
    variant: str
    executed_at: datetime
    total_agents: int
    total_ticks: int
    total_events: int
```

**4-2. 타임라인 재생 (SSE 스트림)**

```http
GET /api/v1/simulation/runs/{run_id}/timeline/stream
  ?speed=1x|4x|16x
  &start_tick=0
Accept: text/event-stream
```

응답은 Server-Sent Events로 틱별 프레임이 흘러나옵니다.

```python
class TimelineFrame(BaseModel):
    tick: int
    day_of_week: str
    time_slot: str
    active_agents: list[AgentMarker]
    active_spots: list[SpotMarker]
    events_this_tick: list[LiveEvent]

class AgentMarker(BaseModel):
    agent_id: str
    persona_type: str
    location: GeoPoint
    current_action: str               # "browsing_feed" | "teaching" | "resting" ...
    fatigue: float
    social_need: float

class SpotMarker(BaseModel):
    spot_id: str
    skill_topic: str
    status: Literal["OPEN", "MATCHED", "IN_PROGRESS", "CLOSED"]
    location: GeoPoint
    participant_count: int
    capacity: int

class LiveEvent(BaseModel):
    event_type: str
    actor_persona: str
    caption: str                      # "spontaneous가 기타 1:1 스팟을 열었어요 (5,000원)"
    importance: float                 # 0~1, UI 하이라이트 여부
```

**4-3. 하이라이트 클립**

```http
GET /api/v1/simulation/runs/{run_id}/highlights
```

```python
class HighlightClip(BaseModel):
    clip_id: str
    title: str                        # "3번째 만남, 친구가 되다"
    category: Literal["first_success", "bond_upgrade", "counter_offer", "referral", "unexpected_match"]
    start_tick: int
    end_tick: int
    involved_agents: list[str]
    narrative: str                    # LLM 생성 3~5 문장
    share_url: str                    # 공유용 정적 페이지
    thumbnail_url: str
```

#### 프론트 사용 예

```ts
// 공개 페이지 진입 시
const runs = await api.get('/v1/simulation/runs?variant=weekend_peak&limit=5');
carousel.render(runs);

// 재생 시작
const eventSource = new EventSource(
    `/v1/simulation/runs/${runId}/timeline/stream?speed=16x`,
);
eventSource.onmessage = (msg) => {
    const frame: TimelineFrame = JSON.parse(msg.data);
    mapLayer.updateAgents(frame.active_agents);
    mapLayer.updateSpots(frame.active_spots);
    frame.events_this_tick
        .filter((e) => e.importance > 0.7)
        .forEach((e) => tickerBar.push(e.caption));
};

// 하이라이트 공유
const highlights = await api.get(`/v1/simulation/runs/${runId}/highlights`);
highlights.forEach((clip) => shareGrid.add(clip));
```

#### 구현 메모

- `event_log.jsonl`을 그대로 스트림하지 않음. 서버가 **tick 단위로 재구성**한 `TimelineFrame`을 캐시(Redis)에 미리 빌드해두고 서빙.
- 하이라이트는 Stage 3의 매력 신호 중 특정 조합(예: `bond_upgrade` 이벤트 + `avg_satisfaction ≥ 0.85`)에 해당하는 구간을 잘라낸 것.
- `narrative`만 LLM 호출(클립당 1회). 나머지는 전부 결정론적.

## 5. 이 피쳐를 논문에 녹이는 방법

### 5-1. 논문의 포지셔닝 한 문단 (초록/서론용)

> _"신규 서비스의 cold-start 문제를 해결하기 위해, 우리는 에이전트 기반 시뮬레이션과 LLM 텍스트 생성을 결합한 **2단 합성 콘텐츠 파이프라인**을 제안한다. 핵심 아이디어는 **'사실(fact)' 생산은 결정론적 시뮬레이션에 맡기고, 'LLM은 그 사실을 문체로 옮겨 쓰는 문서 생성기로 격하시키는 것'**이다. 이를 통해 LLM hallucination을 사실 수준에서 차단하고, 다층 검증 파이프라인(스키마·룰·교차참조·LLM critic·다양성·스코어링)을 통과한 콘텐츠만 서비스 DB에 publish한다. 우리의 적용 도메인은 또래 강사(peer-instructor) 마켓플레이스이며, 50 에이전트 × 7일 시뮬레이션으로 약 1,500개의 스팟 로그를 생성하고, 그 중 매력도·다양성 기준으로 선별된 스팟에 대해 5종 콘텐츠(피드·상세·플랜·메시지·리뷰)를 자동 생성했다. 평가 결과, 1차 승인률 ≥ 70%, 최종 승인률 ≥ 95%, 평균 품질 점수 ≥ 0.80을 달성했다."_

### 5-3. 논문 기여점 3가지 (contribution claim)

1. **사실-문체 분리 원칙**: LLM에게 사실 생산을 맡기지 않고, 결정론적 ABM이 사실을 만든 뒤 LLM은 문서 생성기로만 쓰는 분업 구조.
2. **도메인 특화 페르소나·자산·관계 시뮬레이터**: 또래 강사 마켓플레이스의 핵심 다이나믹스(단골 관계, 용돈 동기, 실비 분리 가격 구조, 역제안, 양방향 origination)를 공식으로 표현.
3. **6-Layer 검증 파이프라인**: LLM 출력의 사실성·스타일·다양성을 각각 다른 층에서 검증하는 defense-in-depth 구조.

### 5-4. 평가 지표 (논문 Experiments 절)

**시뮬레이터 측 (재현성·분포 검증)**

- 스킬 토픽 분포 상위 5개 비중: 60~80%
- 관계 전이율: first_meet → regular ≥ 15%, regular → mentor_bond ≥ 5%
- Fee 분포: peer_labor ∈ [2,500, 10,000] 범위 ≥ 95%, total soft_cap ≤ 15,000 ≥ 80%
- 호스트 수익 분포: 상위 30% 호스트가 전체 수익의 60% (long tail)

**콘텐츠 파이프라인 측 (품질 검증)**

- 1차 승인률 ≥ 70%
- 최종 승인률 ≥ 95%
- 평균 quality score ≥ 0.80
- Diversity ≤ 0.60 (n-gram 중복 상한)
- LLM 호출 횟수 ≤ 15 / 스팟

**육안 검증 (human eval)**

- Live 모드로 3 스팟 × 5 콘텐츠 = 15건 생성 후 저자 3인이 "또래 강사 톤 여부" 평가.

## 7. 자주 나오는 오해 FAQ

**Q1. "그냥 ChatGPT한테 가짜 후기 써달라고 하면 되는 거 아닌가요?"**
A. 그러면 (1) 지역·가격·인물 간 일관성이 깨지고, (2) 서로 다른 후기가 모두 비슷비슷한 평균 문체로 수렴하며, (3) "3번째 수업인데 오늘 처음 친구 됐어요" 같은 **관계 역사에 기반한 내용**이 나올 수 없습니다. 우리 파이프라인은 이걸 시뮬레이션의 `session_count`·`FRIEND_UPGRADE` 이벤트로 강제합니다.

**Q2. "시뮬이 현실이랑 맞는지 어떻게 아나요?"**
A. 100% 맞추는 게 목표가 아닙니다. "**행동 분포가 극단적으로 치우치지 않고, 다양한 시나리오를 만들어낼 수 있는가**"가 기준입니다. Phase F 게이트(§10-1)에서 분포 지표 11개로 이걸 검증합니다.

**Q3. "페르소나 5개면 너무 적은 거 아닌가요?"**
A. MVP 기준입니다. 페르소나는 **YAML anchor 패턴**으로 설계되어, 파일 하나만 떨어뜨리면 추가 가능합니다. 논문에서는 "확장 가능하다"고 명시하고, 실제 실험은 5~7개로 충분합니다.

**Q4. "legacy 라는 말이 자꾸 나오는데 뭔가요?"**
A. 이 프로젝트는 원래 venue 기반(식당/카페/공원 모임) 시뮬이었는데, **또래 강사 모델로 pivot** 한 상태입니다. 기존 코드는 삭제하지 않고 `legacy` flag로 보존했습니다. 논문에서는 peer 모델만 다루면 됩니다.

**Q5. "LLM 비용은 어떻게 감당하나요?"**
A. 사용자 개인의 **Codex 구독**을 subprocess로 호출합니다. 연구 단계에서는 비용 0. 상용화 시점에 API 기반으로 교체 예정.

**Q6. "결과가 진짜 유저한테 먹힐지 어떻게 아나요?"**
A. 이 논문의 **한계 섹션**에 솔직하게 넣어야 할 부분입니다. 현재는 (a) 분포 지표 정량 검증, (b) 저자 3인 육안 검증까지만 했습니다. 실제 유저 A/B 테스트는 future work로 명시합니다.

## 8. 용어집

| 용어                                      | 뜻                                                           | 어디서 왔나                              |
| ----------------------------------------- | ------------------------------------------------------------ | ---------------------------------------- |
| **Cold-start problem**                    | 신규 유저·아이템·서비스 초기에 추천할 데이터가 없는 문제     | 추천 시스템 고전 문제 (Schein 2002)      |
| **Agent-Based Model (ABM)**               | 각 개체를 독립 객체로 두고 상호작용을 시뮬하는 모델링 기법   | 사회과학·경제학 시뮬 계보 (Epstein 1996) |
| **Persona**                               | 사용자 유형을 요약한 가상 캐릭터 프로파일                    | UX / 대화 시스템 (Zhang 2018)            |
| **Tick**                                  | 시뮬레이션 시간 단위. 우리는 1 tick ≈ 반나절                 | Discrete-event simulation 기본           |
| **Hallucination**                         | LLM이 사실이 아닌 것을 그럴듯하게 지어내는 현상              | LLM 평가 연구                            |
| **LLM-as-a-Judge**                        | LLM에게 다른 LLM의 결과를 평가하게 하는 기법                 | Zheng 2023                               |
| **Self-Refine / Rejection feedback loop** | 생성 → 피드백 → 재생성 반복                                  | Madaan 2023, Shinn 2023                  |
| **Defense in depth**                      | 여러 층의 방어를 겹쳐 쓰는 보안 전략 → 여기선 검증 층에 비유 | 보안 공학 용어                           |
| **MMR**                                   | 관련성과 다양성을 동시에 고려한 선별 알고리즘                | Carbonell 1998                           |
| **Publish / Dataset versioning**          | 검증 통과한 콘텐츠만 서비스 DB에 공식 반영하는 절차          | 데이터 엔지니어링 관행                   |

## 10. 한 페이지 슬라이드 요약 (발표용)

```
제목: 또래 강사 마켓플레이스를 위한 합성 콘텐츠 파이프라인

문제: 신규 플랫폼 = 유저 0명 = 피드 비어 있음 = 이탈
기존 해결책: 운영 수기 / LLM 직접 생성 → 확장성·일관성 실패

우리 접근: "사실은 시뮬이, 문체는 LLM이"
┌─ Stage 1: 실제 지역 데이터 → 지역 피처 벡터
├─ Stage 2: 50 에이전트 × 페르소나 × 스킬 × 자산 × 관계 시뮬
│           → event_log 수만 건
├─ Stage 3: N회 시뮬 후 매력 신호 기반 선별 (~1,500 스팟)
└─ Stage 4: LLM으로 5종 콘텐츠 생성 + 6-Layer 검증

기여:
  1. 사실-문체 분리 원칙
  2. 도메인 특화 ABM (관계·자산·실비·역제안)
  3. 6-Layer defense-in-depth 검증

제품 활용:
  1. 가상/현실/혼합 토글 맵 (취미 탐색)
  2. "이 스팟 직접 요청" 전환 CTA (+ 시뮬 미리보기)
  3. 호스트용 스팟 매력도 리포트
  4. 페르소나 라이브 관전 모드 (바이럴)

결과:
  - 1차 승인률 70%+ / 최종 95%+ / 품질 0.80+
  - 재현 가능 (시드 고정)
```
