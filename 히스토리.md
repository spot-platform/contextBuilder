## 1. 구현 방식 (방법론)

이 프로젝트는 "LLM에게 그냥 구현시켰다"가 아니라 **여섯 단계의 의도적 파이프라인**을 거쳐 만들어졌습니다. 새로운 피쳐를 추가할 때도 이 6단계를 재활용하면 동일한 품질을 유지할 수 있습니다.

### 1-1. 여섯 단계 파이프라인

```
[Step 1] 대면 회의 → 자료조사
[Step 2] Claude 앱(브라우저)에 자료 학습
[Step 3] GPT 심층 리서치(Deep Research)로 구현 아이디어 검증
[Step 4] 두 LLM 간 크로스체크 → md 플랜 확정 (에이전트 기반임을 명시)
[Step 5] Claude Code에 플랜 전달 → "먼저 하네스부터" (바로 구현 금지)
[Step 6] 계층별 투두 → 통과할 때까지 반복
```

#### Step 1 — 4월 13일 대면 회의

팀원들과 오프라인 미팅에서 도메인 지식, 추천 시스템의 cold-start 문제, 또래 강사 마켓플레이스의 톤을 정리. 이 회의록이 이후 모든 단계의 "씨앗 문서"가 됨.

#### Step 2 — Claude 앱(웹)에 학습

회의 자료·경쟁 서비스 분석·추천 시스템 기초 문헌을 Claude 웹 앱에 업로드하여 **프로젝트 이해도를 모델 쪽에 주입**. 이렇게 하면 같은 Claude 계정의 후속 대화에서 프로젝트 컨텍스트가 이어짐.

#### Step 3 — GPT 심층 리서치(Deep Research)

구현하고 싶은 피쳐를 GPT Deep Research에 던져 **학술·기술 검증** 을 받음. 여기서 나온 핵심 아이디어가 **3단계 계층 구조** (local-context-builder → spot-simulator → synthetic-content-pipeline).

#### Step 4 — 두 LLM 크로스체크

GPT의 구현 계획을 다시 Claude에게 보여주고 "문제점·보완점을 찾아라"고 요청. 두 모델이 서로를 검증하는 구조. 합의된 초안을 **`.md` 플랜 파일**로 출력하게 함. 이 시점에 "**이 프로젝트는 Claude Code의 agent/harness로 구현할 것**" 을 미리 선언해두어, 플랜이 에이전트 역할 분담 표를 포함하도록 유도.

산출물 예:

- `local-context-builder-plan.md`
- `spot-simulator-implementation-plan.md`
- `spot-simulator-peer-pivot-plan.md` (pivot 후)
- `synthetic_content_pipeline_plan.md`

#### Step 5 — Claude Code에 플랜 전달, 단 "바로 구현하지 마라"

Claude Code의 첫 프롬프트는 항상 "**이 플랜을 읽고 먼저 agent/skill 하네스를 만들어라**"였음. 구현보다 **하네스 설계가 먼저**. 핵심 원칙:

> **"계층마다 새로운 팀을 구성한다."**

즉 infra·schema·collector·processor·qa 다섯 층이면 **층마다 2~5명의 전담 에이전트**를 정의. 같은 에이전트가 모든 층을 하지 않음. 이유: 층별로 책임·도메인 언어가 달라서, 한 에이전트에게 다 맡기면 컨텍스트 오염이 일어남.

실제 생성된 에이전트 (`.claude/agents/`):

- **local-context-builder 팀**: `infra-architect`, `schema-designer`, `collector-engineer`, `processor-engineer`, `integration-qa`
- **spot-simulator 팀**: `sim-infra-architect`, `sim-model-designer`, `sim-engine-engineer`, `sim-data-integrator`, `sim-analyst-qa`
- **synthetic-content-pipeline 팀**: `pipeline-infra-architect`, `codex-bridge-engineer`, `content-generator-engineer`, `validator-engineer`, `pipeline-qa`

필자는 아래의 하네스 라이브러리를 애용 중:
https://github.com/revfactory/harness/blob/main/README_KO.md

#### Step 6 — 계층별 투두 & 반복

각 층의 "완성 기준"을 Claude Code에게 투두 리스트로 먼저 생성시키고, 그 리스트가 **전부 체크될 때까지 재수정**. 완성되면 `_workspace/XX_layer/layer_XX_complete` 같은 마커 파일을 남겨 다음 층이 시작 가능함을 표시.

---

### 1-2. 왜 이 방식인가

| 단계 | 목적                            | 대안 대비 이점                         |
| ---- | ------------------------------- | -------------------------------------- |
| 1~2  | 도메인 주입                     | LLM이 추측으로 채우는 것 방지          |
| 3    | 아이디어 검증                   | 한 모델의 편향 제거                    |
| 4    | 크로스체크 + 에이전트 사전 선언 | 플랜 자체가 하네스 친화적으로 나옴     |
| 5    | 구현보다 하네스 먼저            | 컨텍스트 오염 방지, 층별 재현성 확보   |
| 6    | 투두 + 반복                     | 층 간 회귀 차단, 게이트 기반 품질 관리 |

이 구조의 가장 큰 장점은 **"무엇을 만들었는지"와 "누가 만들었는지"가 층별 파일로 남는다**는 점입니다. `_workspace/XX/README.md`, `_workspace/XX/phaseN_report.md`, `_workspace/XX/*_delta.md` 등을 읽으면 나중에 온 사람이 그 층의 의도와 완성 기준을 되짚을 수 있습니다.

## 2. 수정 방법 (다음에 만질 때)

이 프로젝트를 이어서 수정할 때 **지켜야 할 원칙**은 단 하나입니다.

> **"기존 에이전트와 `_workspace/` 히스토리를 ignore 하지 마라."**

원래 다른 프로젝트에서는 CLI의 도움을 최대한 숨기고 있었는데,
최근 CLI에 대한 생각이 바뀌어서, 오히려 잘 쓰고 있다. 자체도 하나의 스택이라고 생각하기 때문입니다.

### 2-1. 하네스 유지 규칙

새로운 피쳐를 추가할 때 Claude Code에게 던지는 **권장 프롬프트 형식**:

```
[목표] 어떤 피쳐를 추가하고 싶은지 한 줄.

[기존 자산]
- 이 프로젝트의 에이전트 팀 전체(.claude/agents/)를 이그노어하지 말고 함께 써주세요.
- `_workspace/{해당_층}/` 아래 기존 작업 히스토리를 먼저 읽고,
  보완할 수 있는 부분을 찾아주세요.

[요청 형태 A — 보강]
기존 <파일명>을 참고해서, 다음 항목만 추가/수정해주세요: ...

[요청 형태 B — 확장]
기존 하네스의 <에이전트 이름>이 다음 피쳐를 책임지도록 역할을 확장해주세요.
새 에이전트는 추가하지 마세요 (필요 시 사용자와 합의 후).
```

이 형식의 핵심은 세 가지입니다.

1. **에이전트 ignore 금지** — 로컬에만 남아있는 에이전트는 workspace 규칙에 어긋남.
2. **`_workspace` 우선 read** — 이전 층의 의도가 거기 있음.
3. **"보완"과 "확장" 을 구분** — 전면 재작성을 막음.

### 2-2. 수정이 자주 필요한 지점

| 지점            | 파일                                                                                 | 트리거                   |
| --------------- | ------------------------------------------------------------------------------------ | ------------------------ |
| 확률 공식 튜닝  | `spot-simulator/engine/decision.py`, `fee.py`                                        | Phase F 분포 게이트 fail |
| 페르소나 추가   | `spot-simulator/config/persona_templates.yaml`                                       | 시장 변화·실험 요청      |
| 스킬 카탈로그   | `spot-simulator/config/skills_catalog.yaml`                                          | 새 스킬 도메인 오픈      |
| LLM 프롬프트    | `synthetic-content-pipeline/config/prompts/**/*.j2`                                  | 톤 이슈·신규 콘텐츠 타입 |
| 검증 규칙       | `synthetic-content-pipeline/src/pipeline/validators/rules.py`, `config/rules/*.yaml` | 금기어·상한 변경         |
| 스코어링 가중치 | `synthetic-content-pipeline/config/weights/scoring_weights.json`                     | 품질 지표 fail           |

### 2-3. 수정 후 반드시 거쳐야 할 체크

1. **회귀 테스트**: `pytest` — 기존 `tests/`가 전부 green 인지.
2. **층 boundary 점검**: `_workspace/*/boundary_audit.md`의 계약 필드 검증.
3. **분포 재측정**: `spot-simulator/analysis/validate.py` + `validate_peer.py` 게이트 통과.
4. **Live 샘플 1건**: `SCP_LLM_MODE=live` 로 1 스팟만 실제 codex 호출 — 톤 육안 확인.
5. **`_workspace/` 업데이트**: 해당 층의 `*_delta.md`에 이번 변경 요약을 덧붙임. 삭제 금지.

### 2-4. 환경 설정 (.env 및 Codex 연결)

수정하기 전에 **환경을 먼저 재현**해야 합니다. 이 섹션은 두 서브시스템(`local-context-builder`, `synthetic-content-pipeline`)을 처음부터 돌려볼 때 필요한 env 변수·Codex CLI 연결 절차를 정리합니다.

#### 2-4-1. `local-context-builder/.env`

이 서브시스템은 Postgres + Redis + Kakao Local API 를 쓰는 배치 서비스입니다. env 키는 `app/config.Settings` 와 1:1 매핑되며, 계약은 `_workspace/01_infra/env_schema.md` 가 정본입니다.

```ini
# local-context-builder/.env

# 배치 서비스 DB (read/write, 이 프로젝트가 소유)
# psycopg v3 드라이버 사용: postgresql+psycopg://
DATABASE_URL=postgresql+psycopg://lcb:lcb@postgres:5432/lcb

# 실서비스 DB (Spring Boot 쪽, read-only)
# 로컬은 빈 값으로 두면 merge_real_data 잡이 자동 skip.
# 값이 있으면 read-only + statement_timeout 강제.
REALSERVICE_DATABASE_URL=
REALSERVICE_STATEMENT_TIMEOUT_MS=30000

# Celery broker / result backend
REDIS_URL=redis://redis:6379/0

# Kakao Local API 키 (배치 서비스 전용 발급본)
KAKAO_REST_API_KEY=replace-me-with-kakao-rest-key

# Admin API 키 (X-Admin-Key 헤더 검사)
ADMIN_API_KEY=replace-me-with-random-hex

# 현재 배치 대상 도시
TARGET_CITY=suwon

# Postgres 컨테이너 부트스트랩 (docker-compose 가 읽음)
POSTGRES_USER=lcb
POSTGRES_PASSWORD=lcb
POSTGRES_DB=lcb
```

**키 설명**

| 키                                 | 필수        | 소비자                                   | 설명                                                                   |
| ---------------------------------- | ----------- | ---------------------------------------- | ---------------------------------------------------------------------- |
| `DATABASE_URL`                     | O           | `app/db.py`, `migrations/env.py`         | 배치 서비스 전용 Postgres. 반드시 `postgresql+psycopg://` 드라이버     |
| `REALSERVICE_DATABASE_URL`         | X           | `app/db_readonly.py`                     | 실서비스 DB 읽기 전용 커넥션. 로컬은 빈 값 → `merge_real_data` 잡 skip |
| `REALSERVICE_STATEMENT_TIMEOUT_MS` | X           | `app/db_readonly.py`                     | libpq `options` 로 statement timeout 주입 (기본 30s)                   |
| `REDIS_URL`                        | O           | `app/celery_app.py`                      | Celery broker + result backend 겸용                                    |
| `KAKAO_REST_API_KEY`               | O           | `app/collectors/kakao_local_client.py`   | 카카오 REST 키. 발급 후 하드코딩 금지                                  |
| `ADMIN_API_KEY`                    | O           | `app/api/admin.py`                       | FastAPI admin 라우터의 `X-Admin-Key` 가드. constant-time 비교          |
| `TARGET_CITY`                      | X           | `jobs/*`, `collectors/*`, `processors/*` | 현재 배치 스코프 (기본 `suwon`)                                        |
| `POSTGRES_USER/PASSWORD/DB`        | O (compose) | `docker-compose.yml` 만                  | Python 앱은 이 값을 직접 읽지 않음. `DATABASE_URL` 과 수동 동기화      |

**불변식 (어기면 integration-qa 가 reject)**

1. `Settings` 필드와 `.env.example` 키는 1:1. 한쪽에만 추가 금지.
2. 시크릿(`KAKAO_REST_API_KEY`, `ADMIN_API_KEY`, `REALSERVICE_DATABASE_URL`)은 하드코딩 금지. `.env.example` 에는 `replace-me-*` 같은 placeholder 만.
3. `REALSERVICE_DATABASE_URL` 만 유일하게 optional. 모든 코드 경로는 `get_readonly_engine()` 반환이 `None` 일 수 있음을 핸들.
4. `ADMIN_API_KEY` 는 반드시 constant-time 비교.

**기동 절차**

```bash
cd local-context-builder
cp .env.example .env          # 그 후 KAKAO/ADMIN 키만 본인 값으로 교체

docker compose up -d postgres redis
# (Postgres 가 ready 될 때까지 약 5~10s)

alembic upgrade head
uvicorn app.main:app --reload --port 8000
# → http://localhost:8000/docs (admin API)
celery -A app.celery_app worker --loglevel=info   # 별도 터미널
```

설정은 `functools.lru_cache` 로 메모이즈되어 있어 런타임에 `.env` 를 바꿔도 반영되지 않습니다. 테스트에서 새 설정을 주입할 땐 `get_settings.cache_clear()` 후 해당 모듈을 재임포트하세요(`app/db.py`, `app/celery_app.py` 가 import 시점에 설정을 붙잡음).

#### 2-4-2. `synthetic-content-pipeline` — Codex CLI 연결법

이 서브시스템은 **OpenAI/Anthropic API 를 쓰지 않습니다**. 사용자의 **ChatGPT 구독에 딸린 `codex` CLI 를 subprocess 로 호출**합니다. 이게 이 프로젝트에서 가장 비정통적인 부분이니 절차를 정확히 따라야 합니다.

##### 사전 조건: Codex CLI 설치·로그인

```bash
# 1. codex CLI 설치 (OpenAI 공식 배포판 사용)
#    설치 경로는 OpenAI 공식 안내를 따름. 결국 PATH 에 `codex` 가 있어야 함.
which codex
codex --version

# 2. 내 ChatGPT 구독 계정으로 로그인
codex login
#    → 브라우저가 열림. OAuth 완료 후 토큰이 ~/.config/codex/ 아래에 저장됨.

# 3. dry-run 으로 실제 호출 가능 여부 확인 (한 번만)
echo '안녕' | codex exec --json --ephemeral --sandbox read-only -
#    → 응답 JSON 이 stdout 에 오면 OK. 에러가 뜨면 그 메시지 그대로 로그인/권한 점검.
```

**중요**: 파이프라인 코드(`src/pipeline/llm/codex_client.py`)가 호출하는 정확한 커맨드는 아래와 같습니다. 직접 `subprocess.run(["codex", ...])` 을 다른 모듈에서 호출하는 건 금지되어 있고, 반드시 `codex_client` 를 통해야 합니다(향후 `scripts/lint_no_api.py` 가 감시 예정).

```bash
codex exec --json --skip-git-repo-check --ephemeral --sandbox read-only \
  --output-schema <SCHEMA_PATH.json> \
  -o <OUTPUT_MSG.json> \
  [-m <MODEL_NAME>] \
  "<RENDERED_PROMPT>"
```

- `--ephemeral --sandbox read-only`: 우리 프로세스가 파일 시스템을 만지지 못하게 격리.
- `--output-schema`: JSON Schema 강제. 위반하면 `CodexSchemaError` 가 남.
- `-o`: 최종 메시지를 파일로 받음. stdout 은 진행 이벤트용.
- `-m` 은 옵션. ChatGPT 구독은 특정 모델명(`gpt-5-codex` 등) 을 거부할 수 있으므로 비워 두면 CLI default 에 맡김.

##### 파이프라인 측 env 변수

```bash
# synthetic-content-pipeline/.env 또는 셸 export

# LLM 호출 모드
```

##### 모드별 동작 요약

| `SCP_LLM_MODE` | 프롬프트 렌더 | subprocess        | 소스                                                                          | 용도                       |
| -------------- | ------------- | ----------------- | ----------------------------------------------------------------------------- | -------------------------- |
| `stub` (기본)  | skip          | skip              | `tests/fixtures/codex_stub/<dir>/v<n>/<hash[:8]>.json`, 없으면 `default.json` | CI · 회귀 · 무료 개발      |
| `live`         | Jinja2 렌더   | 실제 `codex exec` | 첫 호출 후 캐시 저장 → 다음 호출은 캐시                                       | 최종 품질 검증 · live 샘플 |

##### 기동 절차 — 처음 돌려보는 순서

```bash
cd synthetic-content-pipeline
python -m venv .venv && source .venv/bin/activate
pip install -e .

# 1. 의존성 & 회귀 테스트 (전부 stub 모드)
pytest -q
#   → 153 passed 목표 (Phase F 기준, 기존 baseline)

# 2. stub 모드로 1 스팟 build → generate → publish 한 번
SCP_LLM_MODE=stub python -m pipeline.run --limit 1
#   → ./pipeline.db 에 synthetic_* row 생성 확인

# 3. live 모드로 단일 스팟 smoke
export SCP_LLM_MODE=live
export SCP_LLM_CACHE=off           # 새 응답을 강제로 받아보고 싶을 때만
python -m pipeline.run --limit 1
#   → codex 호출 실패 시 stderr 마지막 30 줄이 CodexCallError 에 담김
```

##### Codex 호출이 실패할 때 진단 순서

1. `which codex && codex --version` — PATH·버전 확인.
2. `codex login` 재실행 — 토큰 만료 가능.
3. 위 "dry-run" 한 줄로 CLI 자체 정상 여부 확인.
4. `CodexRateLimitError` 면 `SCP_CODEX_CONCURRENCY` 를 2~3 으로 내리고 재실행.
5. `CodexSchemaError` 면 해당 스키마 파일(`src/pipeline/llm/schemas/*.json`) 과 프롬프트 템플릿(`config/prompts/<dir>/v<n>.j2`) 의 필드명을 대조.
6. `CodexTimeoutError` 면 `SCP_CODEX_TIMEOUT` 을 300~600 으로 올림. 프롬프트가 너무 길어졌을 가능성.

##### 주의사항

- **절대 `openai` / `anthropic` SDK 를 import 하지 말 것.** 이 프로젝트는 구독 기반 CLI 호출을 정체성으로 삼습니다. API 키를 env 에 넣지 마세요.
- `codex exec` 은 내부적으로 사용자의 `~/.config/codex/` 에 있는 OAuth 토큰을 씁니다. 서버/CI 에 올릴 땐 이 경로 보관·갱신 전략을 별도로 설계해야 합니다(현재는 로컬 개발 전용).
- 캐시는 `(template_id + variables 해시 + schema)` 키로 파일 기반 저장됩니다. 프롬프트 템플릿을 수정했다면 캐시를 비우거나 `SCP_LLM_CACHE=off` 로 강제 재호출하세요.

### 2-5. 협업 규칙 — 이슈 → 브랜치 → PR → 리뷰

이 프로젝트는 혼자 만든 게 아니라 **여러 명이 이어서 만질 자산**입니다. 팀이 커질수록 "누가 언제 왜 어떤 변경을 했는가"를 찾기 어려워지므로 GitHub Flow 의 단순 버전을 강제합니다. 아래 4 단계는 예외 없이 지켜집니다.

#### 2-5-1. 4 단계 워크플로

```
[1] 이슈 생성
    ↓
[2] 이슈 번호로 브랜치 생성 (feature/2, refactor/3, bug/4, docs/5)
    ↓
[3] 작업 → 커밋 → 푸시 → PR 생성 (요약 필수)
    ↓
[4] 최소 1 명 이상 리뷰 승인 후 merge
```

#### 2-5-2. 이슈 먼저 — "이슈 없이 코드 없다"

**원칙**: 어떤 변경이든 먼저 이슈를 만든다. 한 줄 버그 수정도 예외 없음. 이유는 세 가지.

1. **검색성**: 6 개월 뒤에 "왜 이렇게 바꿨더라?" 를 찾는 기준점이 이슈 번호다.
2. **사전 합의**: 코드 쓰기 전에 의도·범위를 글로 한 번 적게 강제. 잘못된 방향의 구현 자체를 줄인다.
3. **브랜치·PR 연결**: 이후 모든 브랜치·PR·커밋이 이 이슈에 묶여 타임라인이 자동 생성된다.

**이슈 작성 템플릿**

```markdown
## 배경 / 문제

(1~3 문장. 왜 이 작업이 필요한가)

## 제안

(무엇을 할 것인가. 1 줄)

## 완료 기준 (Definition of Done)

- [ ] ...
- [ ] 회귀 테스트 pass
- [ ] `_workspace/{해당 층}/*_delta.md` 갱신

## 영향 범위

- 수정되는 파일/모듈:
- 관련 에이전트: `sim-engine-engineer`, `validator-engineer` ...
- 기존 계약 변경 여부: yes / no

## 참고

- 관련 이슈: #12
- 관련 플랜 섹션: `spot-simulator-peer-pivot-plan.md §3-4`
```

**라벨 규칙**: `feature`, `refactor`, `bug`, `docs`, `chore`, `question`. 라벨이 바로 브랜치 prefix 로 이어진다.

#### 2-5-3. 브랜치 네이밍 — 이슈 번호를 그대로 박는다

```
<type>/<issue_number>[-<short-slug>]
```

- `type` 은 이슈 라벨과 동일: `feature`, `refactor`, `bug`, `docs`, `chore`.
- `issue_number` 는 GitHub 이슈 번호.
- `short-slug` 는 선택. 3~5 단어 하이픈 케이스. 생략해도 됨.

**예시**

| 이슈                                | 브랜치                       |
| ----------------------------------- | ---------------------------- |
| #2 "peer_tone_fit 가중치 조정"      | `feature/2-peer-tone-weight` |
| #3 "fee_breakdown 계산 로직 리팩터" | `refactor/3-fee-breakdown`   |
| #4 "영어 프리토킹 combo 쏠림 버그"  | `bug/4-english-combo-skew`   |
| #5 "히스토리 문서 초안"             | `docs/5-history-doc`         |

브랜치를 만들기 전에 항상 `main` 기준 최신 상태에서 fork.

```bash
git fetch origin
git switch -c feature/2-peer-tone-weight origin/main
```

#### 2-5-4. PR — "리뷰어가 5 분 안에 이해할 수 있는 설명" 이 필수

**큰 PR 은 사실상 리뷰되지 않는다.** 500 라인을 넘으면 승인률·결함 발견률이 급락한다는 게 업계 정설. 그래서 두 가지를 강제한다.

1. **분할**: PR 당 300~400 라인 목표. 넘으면 이슈를 쪼개서 PR 을 나눈다.
2. **요약**: PR description 에 아래 템플릿을 반드시 채운다.

**PR description 템플릿**

```markdown
## 이슈

Closes #2

## 요약 (3~5 줄, 리뷰어가 이것만 읽어도 감 잡히게)

- 무엇을 했는가:
- 왜 했는가:
- 어떤 접근을 택했는가 (대안 대비):

## 변경 범위

- 새 파일: ...
- 수정 파일: ...
- 삭제 파일: ...
- 계약 변경: yes / no (yes 면 어떤 필드가 어떻게 바뀌는지)

## 테스트

- [ ] 단위 테스트 추가/수정: 파일 경로
- [ ] `pytest` 전체 pass (회귀 0)
- [ ] 분포 게이트 재측정: `validate_peer.py` 결과 첨부
- [ ] live 샘플 1 건 육안 확인 (해당 시)

## 스크린샷 / 로그 (선택)

(분포 차트, live 샘플 JSON, 에러 before/after 등)

## 리뷰어가 특히 봐줬으면 하는 부분

- ...

## Out of Scope

- 이 PR 에서 다루지 않은 관련 작업 (후속 이슈 번호 기재)
```

**요약 3~5 줄의 목적**: 리뷰어가 diff 를 읽기 전에 "이 PR 이 옳은 방향인지" 만 먼저 판정할 수 있게 하는 장치. 요약만 읽고 "접근이 잘못됐다" 고 느끼면 diff 를 한 줄도 안 읽고 돌려보낸다 → 리뷰어·작성자 모두의 시간 절약.

#### 2-5-5. 리뷰 — 최소 1 명 승인

- **리뷰어 1 명 이상 approve 후 merge**. 본인 merge 금지.
- 리뷰어는 **24 시간 내 1 차 응답** 을 원칙으로 한다. 응답이 코멘트든 approve 든 request-changes 든 상관 없음. 응답이 없으면 작성자가 slack/DM 으로 ping.
- 리뷰 코멘트는 **"blocker / suggestion / nit"** 3 단계로 명시.
    - `[blocker]` — 이 부분이 고쳐지지 않으면 approve 안 함.
    - `[suggestion]` — 더 좋을 것 같지만 강제는 아님. 작성자 재량.
    - `[nit]` — 코드 스타일·명명 같은 사소한 것. 다음 PR 로 미뤄도 됨.
- **리뷰어가 직접 고치지 마라.** PR 작성자의 맥락을 존중. 제안만 남기고 수정은 작성자가.

### 3-2. [토큰 낭비 후보] 역제안(counter-offer) 플로우 추가

**요청 배경**: 세션 3의 사용자 프롬프트 [014] —

> _"혹시 역제안에 대한 플로우도 넣을 수 있을까? 5명 모집이었는데 3명만 모이면 '3분이서 더 내고라도 진행할까요?'라고 호스트가 역제안하는 거."_

**구현 결과**:

- `spot-simulator/engine/negotiation.py`에 `send_counter_offer` / `recompute_fee_for_smaller_group` / `p_accept_counter_offer` / `finalize_counter_offer` 추가.
- 이벤트 4종: `COUNTER_OFFER_SENT`, `COUNTER_OFFER_ACCEPTED`, `COUNTER_OFFER_REJECTED`, `SPOT_RENEGOTIATED`.
- `_workspace/sim_03_engine/counter_offer_trace.md`에 tick-by-tick trace 1건 저장.

**문제점**:

- Cold-start 해결에 **직접 기여하지 않음**. 신규 유저가 피드를 볼 때 "역제안 후 진행된 수업"이라는 맥락이 과연 첫 인상에 필요한가? 초기 피드엔 "성공 케이스"가 압도적이어야 함.
- 검증 파이프라인의 complexity 만 키움. Phase F 분포 게이트에 `counter_offer 발동률 5~15%` 같은 추가 기준이 생김 → QA 부담 증가.
- 실제 peer event_log 에서 renegotiation 이 일어난 spot 수는 소수, 그 중 완주한 케이스는 더 적어 goldens 에 쓸 샘플이 희소.

**교훈**:

- "제품적으로 재미있는 디테일" 과 "cold-start 에 꼭 필요한 디테일" 은 다르다. 전자는 **v2 로 미뤘어야** 했다.
- 지금 와서 제거할 수는 없음 (append-only). 대신 `simulation_config.yaml` 에서 **비활성 flag 로 꺼두는 옵션**을 둬야 함. (TODO)

**토큰 영향 추정**: 이 플로우 전체 구현 + 테스트 + 문서로 **약 30~80K output tokens** 지출. 세션 3 총 출력(1.71M) 대비 2~5% 수준이지만, 더 중요한 건 "다음 작업의 의사결정 컨텍스트를 오염시켰다"는 점.

### 3-3. [토큰 낭비 후보] offer / request 양방향 origination 중간 편입

**요청 배경**: 세션 3 프롬프트 [015] —

> _"offer와 request로써의 관계를 넣을껀데, 이번 변경사항 보니 중간에 녹여줄 수 있다면 한 번에 하는 게 좋아보여서…"_

**구현 결과**:

- `spot-simulator/models/skill_request.py` 신규.
- `CREATE_SKILL_REQUEST`, `SUPPORTER_RESPONDED`, `REQUEST_EXPIRED` 이벤트.
- `ContentSpec.origination_mode = "offer" | "request_matched"`, `originating_voice`, `originating_request_summary` 필드.
- 프롬프트 v2 에 origination-aware 블록 추가.

**문제점 — 사용자가 직접 말한 의도 벗어남**: 프롬프트 [018] 에서 사용자가 다시 정의함 —

> _"request는 서포터를 찾는 피드인거야… 금액이 너무 적어, 알려주는 입장에서 마진이 남아야…"_
>
> 프롬프트 [019] 에서 본인이 자문자답:
> _"이런 고도화가 우리가 원하던 **콜드 스타트, 피드 생성 템플릿, 새 취미 발견**에 도움이 될까? 아니면 매력적인 피드만 있으면 되잖아?"_

**Phase F QA 결과로 드러난 현실** (`_workspace/scp_05_qa/phase_peer_report.md` §1):

> _"origination_mode=='request_matched' 인 spot 은 29 combo 중 **오직 '영어 프리토킹 / small_group / cafe'** 에서만 존재 (160/163건). 다른 combo 는 100% offer 로 생성됨."_

즉 **시뮬 상으로 request 경로는 거의 작동하지 않았음**. 14 스킬 중 1 스킬에만 편향. 이걸 goldens/live sample 에서는 "영어 1건 + offer extra 1건"으로 대표 샘플링할 수밖에 없었음.

**복구 결정**: 프롬프트 [020] —

> _"그러면 금액적인 피드백이랑 진행 방법 피드백은 콜드 스타트에도 필요하니깐, 이런 부분은 살려서 A로 가자."_

즉 **양방향 origination 의 "가격 제안 / 플랜 도움말" 부분만 살리고, 분기 로직 자체는 실효가 없어도 append-only 원칙에 따라 남겨둠**. 이게 현재 상태.

**교훈**:

- 중간에 새 기능을 "이왕이면 같이 넣자" 는 결정은 **위험**. 분리 구현했으면 나중에 드롭이 쉬웠을 것.
- 분포 편향은 QA 단계에서야 보이기 때문에, **구현 전에 "이 기능이 시뮬 분포에 얼마나 등장할까"를 먼저 추정**하는 걸 습관화해야 함.

**토큰 영향 추정**: 약 60~150K output tokens. 세션 3의 약 4~9%.

### 3-4. [분포 편향] 영어 프리토킹·카페 combo 쏠림

**증상**: Phase F 배치 500 spot 중 request_matched 는 160/163 건이 영어 프리토킹 1 combo. 즉 14 스킬 × 3 모드 × 4 venue ≈ 168 combo 중 **실질적으로 작동하는 origination 분기가 1 개**.

**원인**: `spot-simulator/data/region_features.json` 의 `skill_density_*` 가중치가 특정 페르소나·지역 조합에 쏠려 있었음. `spontaneous` 페르소나가 연무동에서 "영어 프리토킹 learn_appetite" 를 높게 가진 반면, 다른 스킬은 learn_appetite 0.5 를 못 넘는 경우가 많았음.

**부분 완화**: persona_templates.yaml 의 learn_appetite 를 일부 보정. 하지만 **근본 해결은 미완**. sim-data-integrator 에게 `region_features.json` 의 density 산출 로직을 재작성 요청해야 함 (TODO).

**근거**: `synthetic-content-pipeline/_workspace/scp_05_qa/phase_peer_report.md` §1

### 3-5. [운영 이슈] Phase 1 구현 중 Request interrupted 2회

세션 3 프롬프트 [003], [004] 에서 연속 `[Request interrupted by user for tool use]` 이 찍힘. 이건 에이전트가 **자동 승인되지 않은 툴**(예: `Bash`)을 대량 호출하려다 차단된 상황. 이후 사용자가 수동으로 허용하며 진행.

**교훈**:

- Claude Code 초기 permission 모드에서 `.claude/settings.json` 의 `permissions.allow` 에 자주 쓰는 툴 묶음을 미리 등록해두면 flow 가 안 끊긴다.
- 특히 `Bash(git:*)`, `Bash(pytest:*)`, `Bash(python3:*)` 같은 패턴을 프로젝트 시작 직후 허용 추가할 것.

### 3-6. [요약] 시행착오에서 뽑은 불변식 3가지

1. **콜드 스타트에 직접 기여하지 않는 피쳐는 v2로 미룬다.** 재미있는 디테일은 구현을 유혹하지만, QA·분포 부담·컨텍스트 오염까지 합치면 비용이 크다.
2. **분포 편향은 구현 전에 예측한다.** "이 기능이 전체 시뮬 분포에 얼마나 나타날까"를 한 문장으로 적지 못하면 구현을 시작하지 않는다.
3. **append-only 는 원칙이되 `flag 로 끌 수 있게` 한다.** 추가하는 건 쉽지만 되돌리는 건 어렵다. `simulation_config.yaml` 에 feature flag 슬롯을 미리 열어둔다.

### 4-2. 해석

- **실제 생성된 텍스트(output)**: 약 **2.2M 토큰**. 이게 실제로 이 프로젝트에 "쓰여진" 새 글자의 총량입니다. 책 한 권이 약 15~20만 토큰이므로 **책 11~15권 분량**을 기계가 썼습니다.
- **캐시 쓰기(cache_creation)**: 10.9M. 새로운 컨텍스트(파일·플랜·히스토리)를 프롬프트 캐시에 넣은 양.
- **캐시 읽기(cache_read)**: **205.9M**. 이게 압도적입니다. 세션 3 하나에서만 163M 토큰을 캐시로 재사용했습니다. 이게 가능했던 이유는 **`.md` 플랜 파일과 `_workspace/` 히스토리가 반복적으로 로드되며 캐시 히트**가 났기 때문입니다.
- **효율성 시사점**: cache_read/output 비율이 **약 94배** (205.9M / 2.2M). 즉 LLM 은 "1 토큰 쓰기 위해 94 토큰의 컨텍스트를 다시 읽었다". 이 비율이 높다는 건 **플랜·워크스페이스·메모리 재활용 전략이 잘 작동했음**을 의미합니다.

### 4-3. 세션 3가 왜 유독 큰가

세션 3(synthetic-content-pipeline + peer pivot)이 전체의 **78%** 를 차지합니다. 이유:

1. Phase 1~4 정규 + peer pivot Phase A~F = **10개 Phase**
2. 각 Phase 완료 시점에 `_workspace/XX/*_delta.md`, `phaseN_report.md` 를 재생성 → 캐시 쓰기 증가
3. 6 Layer 검증기 + 5 generator + 500 spot 배치 처리 → 코드 라인 수 폭증
4. 시행착오 (§3-2, §3-3) 로 인한 재작성

### 4-4. 토큰 다이어트 가이드 (다음 프로젝트용)

- **md 플랜을 한 번에 크게 쓰지 말 것**. 1500 라인 넘는 플랜 하나보다 200~400 라인 플랜 5개가 캐시 효율이 높음.
- **`_workspace/` 를 층별로 잘 쪼갤 것**. 에이전트가 해당 층만 읽으면 되게 해야 cache hit 효율이 오름.
- **피드백 루프에서 "전체 재작성" 금지**. Edit 기반 수정 유도.
- 시행착오 3-2, 3-3 같은 중도 변경 요청을 피하면 바로 **10~15% 토큰 절감** 가능.

## 6. 프롬프트 분석 — 왜 이 요청이 잘 작동했는가

위 요청이 "전문적 프롬프트" 로 불릴 수 있는 이유를 **6가지 원칙** 으로 분해합니다. 이 원칙들을 익혀두면 Claude Code 에 어떤 작업을 던져도 손실이 적습니다.

### 원칙 1 — 산출물 형태를 먼저 못 박는다

_"새로운 파일로 만들어줘, 이름은 히스토리로"_ — 결과가 **한 파일**임을 선언.

- 효과: 에이전트가 "중간 보고서" 나 "초안 여러 개" 를 만들지 않음. 1 회차 생성으로 수렴.
- 안 하면: 모델이 "계획 먼저 드릴까요?" 같은 메타 질문 루프에 빠짐.

### 원칙 2 — 성격(톤·용도)을 단어로 명시한다

_"인수인계서 느낌"_ — 장르를 선언.

- 효과: 모델이 "기술 보고서" 와 "튜토리얼" 과 "인수인계서" 의 차이를 안다. 톤·문단 길이·강조점이 한 번에 정렬됨.
- 교훈: "문서 만들어줘" 보다 "X 느낌으로" 가 훨씬 효율적.

### 원칙 3 — 데이터 소스를 **경로 수준** 으로 지정한다

_"\_workspace를 참고", "다른 claudeCode 대화 색션을 찾아서"_ — 모델이 어디를 뒤질지 안내.

- 효과: Explore 단계에서 grep 범위가 확 좁아지고, 엉뚱한 디렉토리를 뒤지지 않음.
- 더 전문적으로 쓰려면: 경로 글롭(`_workspace/**/*.md`)과 보조 검색어(`counter_offer|origination`)까지 주면 완벽.

### 원칙 4 — 구체적 사건을 **예시 2~3개** 로 던진다

_"synthetic 구현 중 counter request나 request에서 서포터 매칭 시 offer처럼 운영한다 같은 당장 콜드 스타트에 도움되지 않아, 사실상 토큰 낭비였던 점"_.

- 효과: 모델이 "시행착오가 뭔지 모르겠다" 상태가 아니라 **정답 후보 2개를 들고 수색**하는 상태가 됨. 검색 성공률이 극적으로 오름.
- 이걸 안 하면: 모델이 "에러 로그" 같은 일반적 이슈만 뽑아오고, 의사결정의 turning point 를 놓침.

### 원칙 5 — 메타 작업(토큰 체킹·프롬프트 분석) 을 **같이** 요청한다

_"토큰 수를 채킹 가능하다면 해주고, 이 요청을 베스트 프롬프트로 소개해줘"_.

- 효과: 같은 컨텍스트 안에서 여러 산출물이 나옴 → **캐시 재사용률 상승**. 별도 대화로 쪼개는 것보다 토큰 효율이 좋음.
- 주의: 너무 많은 메타 태스크를 묶으면 한 섹션이 얕아질 수 있음. 3 개 정도가 적정.

### 원칙 6 — "가능하다면" 이라는 **부드러운 조건** 을 넣는다

_"가능하다면 해주고"_.

- 효과: 모델이 "사실 확인 실패 시 거짓 지어내지 않고 스킵" 하도록 허용. hallucination 억제.
- 특히 토큰 수처럼 **실측이 가능한지 모르는** 데이터를 요청할 때 유용.

### 6-7. 이 프롬프트가 한 단계 더 전문적으로 가려면

- **성공 기준**을 한 줄 추가: _"다음에 이 프로젝트를 처음 보는 사람이 30분 안에 수정 지점을 찾을 수 있어야 한다."_
- **길이 상한** 지정: _"최대 1200 라인, 섹션당 1 표 이상."_
- **불허 리스트**: _"이모지 금지, 영어 감탄사 금지, 'Let me ...' 같은 자기서술 금지."_
- **검증 절차**: _"작성 후 `wc -l`, `grep -c '^## '`, `grep -c '^| '` 결과를 보고해라."_

위 네 가지만 추가하면 동일 프롬프트의 재현성이 한 단계 더 올라갑니다.
